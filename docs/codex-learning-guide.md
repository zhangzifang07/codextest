# Codex 从入门到精通中文学习文档

> 面向 Windows + PowerShell 用户。本文档帮助你从零开始使用 Codex CLI 和 Codex Windows 桌面软件，并逐步形成自己的 AI 编程工作流。

## 0. 先说结论：Codex 适合做什么

Codex 是一个面向软件开发的 AI 编程助手。它不是只能“回答问题”的聊天工具，而是可以在你的本地项目里读代码、写代码、运行命令、分析报错、生成计划、审查变更、整理文档，并在必要时向你申请权限。

你可以把 Codex 理解成一个会读仓库、会改文件、会跑测试、会解释代码的结对工程师。

最常见的用途：

- 读懂陌生项目：梳理目录、入口、调用链、配置、数据库脚本。
- 修改代码：新增功能、修 bug、重构、补测试、改文档。
- 审查代码：找潜在缺陷、回归风险、缺失测试。
- 自动化任务：定期检查、批量改造、生成报告。
- 学习项目：让 Codex 用你能听懂的语言解释某个模块。

使用 Codex 的核心习惯：

1. 先让它读项目，不要一上来就让它改。
2. 需求说清楚：目标、范围、限制、期望输出。
3. 改动前要计划，改动后要测试和复盘。
4. 重要项目要写 `AGENTS.md`，让 Codex 记住团队规则。
5. 不要把密钥、密码、账号、生产库连接串直接发给 Codex。

## 2. Windows 环境准备

### 2.1 推荐软件

建议先准备这些工具：

- Windows 10/11。
- PowerShell，Windows 自带即可。
- Git for Windows。
- Node.js LTS。
- VS Code，可选但推荐。
- 一个 OpenAI / ChatGPT 账号，具体登录方式以当前官方说明为准。

检查 Git：

```powershell
git --version
```

检查 Node.js 和 npm：

```powershell
node --version
npm --version
```

如果提示找不到命令，说明还没安装或环境变量没有生效。安装完成后建议重新打开 PowerShell。

### 2.2 PowerShell 常用命令

进入项目目录：

```powershell
Set-Location D:\codextest
```

查看当前目录：

```powershell
Get-Location
```

查看文件：

```powershell
Get-ChildItem
```

递归查看：

```powershell
Get-ChildItem -Recurse
```

查看文件内容：

```powershell
Get-Content .\AGENTS.md
```

### 2.3 Windows 路径写法

Windows 常见路径：

```text
D:\codextest
C:\Users\你的用户名\项目
```

PowerShell 中路径有空格时加引号：

```powershell
Set-Location "D:\my project"
```

给 Codex 描述路径时，最好写完整路径，例如：

```text
请阅读 D:\codextest\src 目录，梳理主要模块。
```

## 3. 安装和启动 Codex CLI

### 3.1 安装 CLI

官方推荐的 npm 安装方式：

```powershell
npm i -g @openai/codex
```

安装后检查：

```powershell
codex --version
codex --help
```

如果 `codex` 命令找不到，常见原因是 npm 全局目录没有加入 PATH。先重新打开 PowerShell；仍不行再检查 Node.js/npm 安装。

### 3.2 启动 Codex

进入项目目录后启动：

```powershell
Set-Location D:\codextest
codex
```

首次启动通常会要求登录或授权。按终端提示操作即可。登录细节可能随版本变化，遇到不一致时以官方文档为准。

### 3.3 CLI 的基本工作方式

你在项目目录中启动 `codex` 后，它会把当前目录当作工作区。之后你可以直接输入任务：

```text
请先阅读这个项目的目录结构，告诉我它是什么类型的项目，暂时不要修改文件。
```

或者：

```text
请帮我修复 tests 里失败的测试。先运行测试并分析原因，再给出最小修改方案。
```

更适合初学者的写法：

```text
我是初学者，请一步一步解释你正在做什么。修改文件前先告诉我计划。
```

## 4. Codex CLI 常用命令和场景

### 4.1 交互模式

最常用：

```powershell
codex
```

适合：

- 读代码。
- 一边问一边改。
- 让 Codex 解释操作。
- 需要你确认权限或方向的任务。

### 4.2 非交互模式

`codex exec` 适合自动化或一次性任务：

```powershell
codex exec "请检查这个仓库的 Markdown 文件是否有明显错别字，并输出报告"
```

适合：

- CI 辅助检查。
- 批量生成报告。
- 每次都执行同一类固定任务。

初学阶段建议先用交互模式，熟悉后再用 `codex exec`。

### 4.3 查看 MCP

在 Codex 中可以查看 MCP 工具状态：

```text
/mcp
```

CLI 也支持 MCP 相关命令，具体以当前 `codex mcp --help` 输出为准：

```powershell
codex mcp --help
```

### 4.4 计划模式

在支持的 Codex 环境中，可以使用计划模式让 Codex 只规划、不改代码：

```text
/plan
```

适合：

- 需求还不清楚。
- 项目很大，怕误改。
- 需要先让 Codex 给出实施方案。
- 想让 Codex 先问清楚范围。

常用提示词：

```text
请进入计划思路：先阅读相关代码，确认当前实现，再给出一个可以交给工程师执行的详细计划。现在不要修改文件。
```

## 5. Codex Windows 桌面软件怎么用

Codex Windows 桌面软件适合不想长期在终端里操作的用户。它通常提供更清晰的对话、文件引用、权限提示、计划展示、图片/文档展示、自动化等能力。

### 5.1 基本使用流程

推荐流程：

1. 打开 Codex Windows 软件。
2. 选择或打开你的项目目录。
3. 输入目标，让 Codex 先读项目。
4. 让 Codex 给出计划。
5. 确认后让它修改文件。
6. 让它运行测试或检查。
7. 查看最终总结。

适合初学者的第一条指令：

```text
请先不要修改文件。帮我阅读这个仓库，说明：
1. 这是一个什么项目
2. 主要目录分别做什么
3. 当前有哪些可运行命令
4. 我作为初学者应该先看哪些文件
```

### 5.2 桌面软件和 CLI 的区别

CLI 优点：

- 更适合工程师和终端工作流。
- 适合脚本化和 `codex exec`。
- 和 Git、测试命令、构建命令结合自然。

Windows 桌面软件优点：

- 更适合学习和可视化协作。
- 权限提示更明显。
- 多线程、多文件、计划、自动化体验更直观。
- 更容易查看 Codex 的修改总结。

建议：

- 初学者先用 Windows 桌面软件理解工作流。
- 熟悉后用 CLI 处理高频开发任务。
- 自动化任务、批处理任务再学习 `codex exec`。

## 6. 权限、沙箱和安全边界

Codex 为了安全，会区分哪些命令能直接运行，哪些命令需要你批准。

常见需要注意的操作：

- 安装依赖：可能需要网络访问。
- 删除文件：风险较高。
- 修改系统目录：通常需要批准。
- 打开 GUI 程序：可能需要批准。
- 访问外部网络：可能受限制。

你可以这样要求 Codex：

```text
任何删除文件、重置 Git、修改系统配置、安装依赖的操作，都必须先征求我的确认。
```

不建议这样说：

```text
不用问我，直接执行所有命令。
```

更安全的写法：

```text
请优先使用只读命令分析问题。需要修改文件或安装依赖时，先说明原因、影响范围和替代方案。
```

## 7. 写好 AGENTS.md

`AGENTS.md` 是给 Codex 和其他 AI 编程代理看的项目说明书。它可以告诉 Codex：

- 项目结构。
- 如何构建和测试。
- 代码风格。
- 哪些目录不能改。
- 提交规范。
- 安全规则。
- 团队偏好。

你的仓库根目录已经有 `AGENTS.md`，这是很好的开始。建议每个重要项目都维护一份。

### 7.1 推荐内容

一个实用的 `AGENTS.md` 通常包含：

- 项目结构。
- 本地启动命令。
- 测试命令。
- 格式化和 lint 命令。
- 命名规范。
- 不允许提交的文件。
- 变更前后需要做的检查。
- 特殊业务规则。

### 7.2 让 Codex 帮你完善 AGENTS.md

提示词：

```text
请阅读当前仓库，帮我检查 AGENTS.md 是否完整。不要直接修改，先列出建议补充的内容，重点关注构建命令、测试命令、代码风格、安全规则和不能修改的目录。
```

确认后再让它修改：

```text
请根据刚才的建议更新 AGENTS.md，保持中文，内容简洁，适合以后让 Codex 读取。
```

## 8. MCP：让 Codex 连接更多工具

MCP 可以理解为“工具连接协议”。通过 MCP，Codex 可以连接文档、数据库、浏览器、内部系统、设计工具、搜索工具等。

常见用途：

- 查官方文档。
- 访问内部知识库。
- 查询数据库结构。
- 控制浏览器做页面检查。
- 读取接口文档。

查看 MCP 帮助：

```powershell
codex mcp --help
```

在 Codex 对话中查看可用 MCP：

```text
/mcp
```

学习建议：

1. 初学时先不用 MCP，把基础读代码、改代码、跑测试练熟。
2. 熟悉后先配置一个文档类 MCP。
3. 再尝试浏览器、数据库、内部系统等高价值工具。
4. 涉及公司内部系统时，要确认权限和数据安全规则。

## 9. Skills：把常用能力沉淀成技能

Skills 可以理解为 Codex 的专项工作说明。例如：

- 生成图片。
- 查 OpenAI 文档。
- 创建插件。
- 安装技能。
- 团队内部专用开发规范。

你可以把反复出现的工作流沉淀成 skill，让 Codex 以后按固定步骤做事。

适合做成 skill 的内容：

- 公司内部项目开发规范。
- 固定的代码审查标准。
- 特定框架的开发流程。
- 文档生成流程。
- 发布检查清单。

提示词：

```text
我经常需要让 Codex 按固定流程分析某类项目。请帮我设计一个 skill 的内容大纲，包括触发场景、操作步骤、注意事项和输出格式。
```

## 10. Subagents：把任务拆给子代理

在支持子代理的 Codex 环境中，可以把任务拆成多个并行子任务。例如：

- 一个代理读前端代码。
- 一个代理读后端代码。
- 一个代理查测试。
- 主线程负责整合结论。

适合使用子代理的场景：

- 大型项目分析。
- 多模块并行排查。
- 一个任务可以拆成互不冲突的部分。
- 需要快速收集多个方向的信息。

不适合使用子代理的场景：

- 很小的单文件修改。
- 需求还没说清楚。
- 多个代理会同时改同一个文件。

提示词：

```text
请把这个问题拆成几个可以并行调查的方向。每个方向说明目标、需要阅读的目录、预期输出。暂时不要改代码。
```

## 11. Automations：自动化和定时任务

在 Codex 桌面环境中，自动化可以用来做提醒、定期检查、持续跟进。例如：

- 每天检查项目测试是否失败。
- 每周生成一次技术债报告。
- 定期提醒你复盘某个学习计划。
- 让当前线程过一会儿继续跟进。

适合自动化的任务：

- 重复出现。
- 输出形式稳定。
- 不需要复杂人工判断。
- 风险较低。

不适合自动化的任务：

- 需要大量人工确认的任务。
- 可能删除或批量修改文件的任务。
- 涉及敏感数据的任务。

提示词：

```text
请帮我创建一个每周一上午检查项目 TODO 和 FIXME 的自动化任务，输出一份简短报告，不要自动修改文件。
```

## 12. 从入门到熟练的练习路线

### 第 1 天：跑通安装和基础对话

目标：

- 安装 Codex CLI。
- 打开 Codex Windows 软件。
- 在一个测试仓库里让 Codex 读目录。
- 学会说“先不要修改文件”。

练习：

```text
请先不要修改文件。阅读当前目录，告诉我这个仓库有哪些文件，每个文件大概做什么。
```

```text
请用初学者能听懂的话解释 AGENTS.md 的作用。
```

### 第 1 周：完成小型开发任务

目标：

- 让 Codex 找入口。
- 让 Codex 给计划。
- 让 Codex 修改一个小功能。
- 让 Codex 运行测试或手工检查。

练习：

```text
请先找到和“用户登录”相关的代码入口，说明调用链。不要修改文件。
```

```text
请基于刚才的分析，给出最小修改方案。要求列出需要改哪些文件、为什么改、怎么测试。
```

```text
请按方案实施。保持改动范围最小，修改后运行相关测试，并总结改动。
```

### 第 2-4 周：掌握项目协作

目标：

- 学会写更好的 `AGENTS.md`。
- 学会让 Codex 审查代码。
- 学会处理测试失败。
- 学会用 Git 管理变更。

练习：

```text
请以代码审查的方式检查当前改动，优先指出 bug、回归风险和缺失测试。没有问题也请明确说明。
```

```text
请运行测试。如果失败，先解释失败原因，再给出修复方案，不要直接乱改。
```

```text
请查看 git diff，总结这次改动适合写成什么 commit message，使用 Conventional Commits 格式。
```

### 第 2 个月：形成自己的工作流

目标：

- 形成固定提示词模板。
- 给常用项目维护 `AGENTS.md`。
- 学会配置 MCP。
- 尝试自动化。
- 能判断什么时候该让 Codex 改、什么时候只让它查。

练习：

```text
请帮我把最近 5 次和 Codex 协作的经验整理成一份团队使用规范，重点写：提问方式、修改前检查、测试要求、安全禁区。
```

## 13. 常用中文提示词模板

### 13.1 读懂项目

```text
请先不要修改文件。阅读当前仓库，按下面结构输出：
1. 项目类型和技术栈
2. 主要目录说明
3. 关键入口文件
4. 构建、测试、启动命令
5. 新手应该先看的 5 个文件
```

### 13.2 找调用链

```text
请围绕“这里写你的功能名或报错现象”查找相关代码，输出：
1. 入口在哪里
2. 调用了哪些关键函数/类
3. 数据从哪里来，到哪里去
4. 可能的风险点
暂时不要修改文件。
```

### 13.3 写实施计划

```text
请基于当前代码给出实施计划。要求：
1. 先说明当前实现
2. 再说明最小改动方案
3. 列出会修改的文件
4. 列出测试方法
5. 标出不确定点
现在不要改代码。
```

### 13.4 修改代码

```text
请按刚才确认的方案修改代码。要求：
1. 改动范围最小
2. 遵循现有风格
3. 不改无关文件
4. 修改后运行相关测试
5. 最后总结改动文件和验证结果
```

### 13.5 查 bug

```text
这个功能出现了下面的问题：
【粘贴报错或现象】

请先定位原因，不要急着修改。输出：
1. 最可能原因
2. 证据来自哪些文件或日志
3. 修复方案
4. 需要补充的测试
```

### 13.6 代码审查

```text
请以 code review 的方式检查当前改动，重点关注：
1. 明显 bug
2. 回归风险
3. 安全问题
4. 缺失测试
5. 命名和可维护性

请按严重程度排序，引用具体文件和行号。
```

### 13.7 写文档

```text
请根据当前实现写一份中文使用文档。要求：
1. 面向初学者
2. 包含使用步骤
3. 包含常见问题
4. 包含示例命令
5. 不包含敏感信息
```

### 13.8 让 Codex 始终中文回复

```text
后续请默认用简体中文回复。遇到英文术语时，先给中文解释，再保留必要的英文原词。
```

你也可以把这条规则写进 `AGENTS.md`。

## 14. Git 协作建议

Codex 很适合配合 Git 使用。推荐流程：

1. 修改前查看状态。
2. Codex 改代码。
3. 查看 diff。
4. 运行测试。
5. 让 Codex 总结。
6. 自己确认后提交。

常用命令：

```powershell
git status
git diff
git diff --stat
```

让 Codex 总结当前改动：

```text
请查看当前 git diff，用中文总结这次改动，并给出一个符合 Conventional Commits 的提交信息。
```

提交信息示例：

```text
docs: add codex learning guide
```

注意：

- 不要让 Codex 随便执行 `git reset --hard`。
- 不要让 Codex 删除你不确定的文件。
- 不要提交 `.env`、密钥、账号、临时日志。

## 15. 测试和验证习惯

让 Codex 修改代码后，最好总是问：

```text
请说明你做了哪些验证。如果没有运行测试，请说明原因，并给出我可以手动执行的测试命令。
```

常见测试命令根据项目不同而不同：

```powershell
npm test
npm run test
npm run lint
npm run build
pytest
mvn test
gradle test
```

如果项目还没有测试命令，可以让 Codex 帮你找：

```text
请阅读项目配置，找出这个项目可能的测试、构建、启动命令。不要修改文件。
```

## 16. 常见问题排查

### 16.1 `codex` 命令找不到

可能原因：

- 没安装成功。
- npm 全局目录没进 PATH。
- 安装后没有重新打开 PowerShell。

建议：

```powershell
npm --version
npm list -g --depth=0
codex --help
```

### 16.2 安装依赖失败

可能原因：

- 网络访问受限。
- npm registry 访问慢。
- 公司代理或防火墙限制。
- Node.js 版本过旧。

建议先让 Codex 分析错误：

```text
下面是安装失败日志。请帮我判断原因，并给出 Windows PowerShell 下的排查步骤。不要让我直接粘贴密钥或账号。
```

### 16.3 Codex 想申请权限

先看它要做什么：

- 是否要安装依赖？
- 是否要访问网络？
- 是否要删除文件？
- 是否要修改系统目录？

可以要求：

```text
请先解释这个权限申请的目的、会影响哪些文件、有没有只读替代方案。
```

### 16.4 Codex 回答太泛

把问题变具体：

```text
请不要泛泛解释。请引用具体文件、函数、配置项，并说明判断依据。
```

### 16.5 Codex 改偏了

立刻收窄范围：

```text
先暂停修改。请对照我的原始需求，列出当前改动中哪些是必要的，哪些可能超出了范围。不要继续改文件。
```

## 17. 什么时候使用 WSL2

如果你主要做 Windows 项目、Java 项目、普通前端项目，PowerShell 通常够用。

考虑 WSL2 的情况：

- 项目原本面向 Linux。
- 构建脚本大量使用 bash。
- 依赖 Linux 工具链。
- Docker/Linux 环境更接近生产。

不建议一开始就折腾 WSL2。先在 Windows + PowerShell 跑通 Codex 基础流程，再根据项目需要决定。

## 18. 最佳实践清单

每次开始任务前：

- 说明目标。
- 说明范围。
- 说明不要改哪些文件。
- 让 Codex 先读代码。

修改前：

- 要求 Codex 给计划。
- 确认要改的文件。
- 确认测试方法。

修改后：

- 查看总结。
- 查看 diff。
- 运行测试。
- 让 Codex 做一次 review。

长期使用：

- 维护 `AGENTS.md`。
- 沉淀提示词模板。
- 给重复任务做自动化。
- 给高频工具配置 MCP。

## 19. 一套推荐日常工作流

你可以每天按这个顺序用 Codex：

1. 打开项目。
2. 让 Codex 读取 `AGENTS.md` 和当前任务相关目录。
3. 让 Codex 输出计划。
4. 确认计划后实施。
5. 运行测试。
6. 让 Codex review 当前 diff。
7. 让 Codex 生成提交信息。
8. 自己最终确认并提交。

完整提示词：

```text
请按下面流程协助我完成任务：
1. 先阅读 AGENTS.md 和相关代码
2. 说明当前实现
3. 给出最小修改计划
4. 经我确认后再改代码
5. 修改后运行相关测试
6. 最后 review 当前 diff，并给出提交信息

我的任务是：这里写你的任务。
```

## 20. 给初学者的最后建议

刚开始不要追求“让 Codex 一次性完成所有事”。你真正要练的是把任务拆清楚：

- 先让它查。
- 再让它计划。
- 再让它改。
- 再让它测。
- 最后让它复盘。

只要你坚持这个节奏，Codex 会从“能帮你写几行代码的工具”，慢慢变成“能陪你理解项目、推进任务、沉淀经验的开发伙伴”。
