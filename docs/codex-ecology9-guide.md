# 使用 Codex 进行泛微 Ecology9 二开指南

## 适用场景
本文档面向使用 Codex 辅助泛微 `Ecology9` 二次开发的场景，重点包括两类工作：
- 快速读懂现有项目结构、模块职责和调用链路。
- 在已有框架中新增页面、接口、流程规则或后台逻辑。

## 一、先让 Codex 读懂项目
开始前，先把 Codex 放在项目根目录，并明确你的目标，例如：

```text
请先帮我梳理这个 Ecology9 项目的目录结构，说明
1. 自定义代码在哪些目录
2. 流程、门户、接口相关代码分别在哪
3. 哪些是标准包，哪些是二开代码
```

建议让 Codex 优先完成以下工作：
- 扫描目录结构，标出 `src/`、`web/`、`WEB-INF/`、`workflow/`、`interface/`、`custom/` 等关键目录。
- 识别核心技术栈，例如 `Java`、`JSP`、`JavaScript`、`XML`、数据库脚本。
- 找出自定义入口，例如自定义 action、service、controller、定时任务、流程节点后操作。
- 梳理从页面到后端再到数据库的调用链。

你可以继续追问：

```text
请按“页面层、接口层、业务层、数据层”整理这个项目，并指出每层的主要文件。
```

## 二、读代码时推荐的提问方式
针对 `Ecology9` 项目，建议把问题问得具体，Codex 才能更快定位：
- “这个审批流程提交时，调用了哪些 Java 类和 SQL？”
- “这个 JSP 页面上的按钮提交后，后端入口在哪个文件？”
- “请分析这个模块依赖了哪些工具类、配置表和流程节点配置。”
- “请区分标准代码和我们公司新增的代码，并说明判断依据。”

如果项目较大，先让 Codex 做局部分析，不要一开始就要求全量解释。

## 三、开发新逻辑的推荐流程
在新增功能前，先让 Codex 基于现有实现找相似案例：

```text
请在项目中找到一个和“流程提交后写业务表”最接近的已有实现，
总结它的入口、参数来源、数据库操作和异常处理方式。
```

然后按下面的顺序推进：
1. 先定位相似模块，复用已有目录结构、命名方式和调用模式。
2. 让 Codex 输出改动方案，明确要新增哪些类、页面、配置或 SQL。
3. 再让 Codex实际修改代码，并限制改动范围，避免误改标准代码。
4. 修改后让 Codex 做一次回查，确认调用链、依赖和影响范围。

推荐直接这样下指令：

```text
请基于现有 XXX 模块，新增一个同风格的功能。
要求：
1. 不修改标准包逻辑，优先放在自定义目录
2. 保持现有命名风格
3. 补充必要注释
4. 说明每个改动文件的作用
```

## 四、Ecology9 二开时重点关注的内容
- 页面改动：确认是 `JSP`、前端脚本还是门户组件在驱动页面行为。
- 流程改动：确认节点前后操作、出口条件、表单字段映射和流程触发点。
- 接口改动：确认是内部接口、第三方集成，还是定时任务/消息回调。
- 数据改动：确认读写的是系统标准表还是自定义业务表，避免误伤主流程。
- 权限改动：确认角色、分部、部门、人员权限是否会影响功能可见性。

## 五、让 Codex 更高效的实践建议
- 一次只处理一个明确目标，例如“找入口”“补逻辑”“查影响范围”。
- 提供具体文件名、目录名、表名、流程编号、字段名，减少泛化分析。
- 修改前先要求 Codex 输出实施方案，修改后再要求它复盘变更。
- 涉及 SQL、流程节点、接口出参时，让 Codex 明确写出假设条件。

## 六、推荐协作模板
可以直接复用下面这段话作为日常指令：

```text
这是一个泛微 Ecology9 二开项目。请先不要直接改代码，先完成下面几步：
1. 梳理相关目录和核心入口
2. 说明当前业务链路
3. 找一个最相近的已有实现
4. 给出最小改动方案
5. 在我确认后再开始改代码
```

如果要直接进入开发，可用：

```text
请基于现有代码完成这个 Ecology9 二开需求：
1. 先定位入口和相似实现
2. 再修改代码
3. 最后列出改动文件、风险点和测试建议
```
