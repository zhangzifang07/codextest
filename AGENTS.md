# 仓库指南

## 项目结构与模块组织
当前仓库还是一个干净的脚手架。请保持根目录简洁，将实现代码放入专门目录：
- `src/`：应用或库代码（按功能/模块分组）。
- `tests/`：自动化测试，目录结构尽量与 `src/` 对应。
- `assets/`：静态资源（图片、示例数据、测试夹具）。
- `docs/`：设计说明、架构决策、使用文档。
- `scripts/`：可复用的开发与 CI 自动化脚本。

示例结构：`src/auth/`、`tests/auth/`、`docs/architecture.md`。

## 构建、测试与开发命令
当前仓库尚未配置构建系统。引入工具后，请提供稳定入口命令，并同步更新本文件和 CI：
- `make setup`（或等效命令）：安装依赖。
- `make lint`：执行静态检查/格式化校验。
- `make test`：运行完整测试套件。
- `make build`：生成发布产物。

在此之前，可先用 PowerShell 做基础检查，例如 `Get-ChildItem -Recurse` 查看仓库内容。

## 代码风格与命名规范
保持模块清晰、职责单一，并统一命名：
- 目录名：小写，必要时使用连字符。
- 源码文件名：遵循语言习惯（如 `snake_case.py`、`kebab-case.ts`）。
- 类/类型：`PascalCase`；函数/变量：按语言约定使用 `camelCase` 或 `snake_case`。
- 优先使用自动格式化与 Lint 工具，避免手工格式不一致。

## 测试规范
每个功能与缺陷修复都应补充测试。`tests/` 中的路径尽量镜像 `src/`，测试名称要具备可读性，例如 `test_login_rejects_invalid_token`。同时覆盖成功路径与失败路径，合并前保证有实质性的覆盖。

## 提交与合并请求规范
当前工作区没有可用 Git 历史，建议从现在开始采用 Conventional Commits：`feat:`、`fix:`、`docs:`、`refactor:`、`test:`。

每个 PR 应包含：
- 变更内容与目的的简要说明。
- 关联的任务/Issue 编号（如适用）。
- 测试证据（命令输出或截图）。
- 涉及 UI 变更时提供截图。

## 安全与配置建议
禁止提交密钥或敏感信息。运行时配置使用环境变量管理；新增配置项时，请提供脱敏的 `.env.example`。
